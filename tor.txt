ข้อ 1  ความเป็นมาและวัตถุประสงค์
1.1 ความเป็นมา
คณะนิติศาสตร์ มหาวิทยาลัยธรรมศาสตร์ มีความประสงค์จะพัฒนาระบบ TULAW ONE PLATFORM เพื่อเป็นศูนย์กลางดิจิทัลแบบ Single Sign-On (SSO) สำหรับบุคลากรของคณะ โดยรวบรวมระบบบริหารจัดการต่าง ๆ ไว้ในแพลตฟอร์มเดียว ครอบคลุมการบริหารบุคลากร เอกสาร โครงการ การสื่อสารภายใน และการจองทรัพยากรร่วมกัน
ปัจจุบันบุคลากรของคณะต้องเข้าถึงระบบงานหลายระบบแยกกัน ส่งผลให้การทำงานไม่คล่องตัว เกิดความซ้ำซ้อนในการจัดการข้อมูล และยากต่อการตรวจสอบกิจกรรมการใช้งานระบบ คณะจึงเห็นสมควรให้มีการพัฒนาระบบรวมศูนย์ดังกล่าว
1.2 วัตถุประสงค์
เพื่อพัฒนาระบบ TULAW ONE PLATFORM ที่รวมระบบงานของคณะไว้ในจุดเดียว รองรับการใช้งานผ่านระบบ Single Sign-On ของ Microsoft
เพื่อเพิ่มประสิทธิภาพการทำงานของบุคลากร ลดขั้นตอนการเข้าถึงข้อมูลและระบบงานต่าง ๆ
เพื่อให้ผู้บริหารมีเครื่องมือสำหรับติดตามและตัดสินใจเชิงนโยบายบนข้อมูล Real-time


ข้อ 2  ขอบเขตของงาน (Scope of Work)
ผู้รับจ้างต้องพัฒนาระบบ TULAW ONE PLATFORM ให้ครอบคลุมโมดูลทั้ง 10 โมดูล โดยมีขอบเขตฟังก์ชันการทำงานดังต่อไปนี้
2.1 โมดูลและขอบเขตฟังก์ชัน
ผู้รับจ้างต้องพัฒนาระบบให้ครบถ้วนตามโมดูลและฟังก์ชันการทำงานที่กำหนด ดังนี้
2.1.1  Dashboard (ศูนย์รวมข้อมูล)
-แสดงข้อมูลสรุปภาพรวมองค์กรแบบ Real-time พร้อมแสดงวันและเวลาที่ข้อมูลซิงค์ล่าสุด
-รองรับ Dashboard แยกรายฝ่ายได้อย่างน้อย 3 ฝ่าย ได้แก่ ฝ่าย IT ฝ่ายวิชาการ และฝ่ายสนับสนุน
-รองรับการแสดงข้อมูลในมุมมอง 5 รูปแบบ ได้แก่ Overview, Weekly, Trend, Proportion และ Comparison
-มีระบบค้นหาขั้นสูง (Advanced Search) รองรับการค้นหาด้วยคำสำคัญ ช่วงเวลา และหมวดหมู่
-แสดงประกาศสำคัญล่าสุดขององค์กรพร้อมลิงก์ดูประกาศทั้งหมดแบบ Real-time

2.1.2  Application Hub (ศูนย์กลางแอปพลิเคชัน)
-รองรับการเชื่อมโยงระบบงานของคณะไว้ในจุดเดียว โดยสามารถจัดหมวดหมู่ระบบงานตามประเภทการใช้งานได้
-แสดงสถิติภาพรวมแบบ Real-time อย่างน้อย 4 รายการ ได้แก่ จำนวนระบบทั้งหมด จำนวนผู้ใช้งาน Active จำนวนระบบออนไลน์ และระบบที่กำลังบำรุงรักษา
-รองรับการปักหมุด (Pin) แอปพลิเคชันที่ใช้บ่อยแยกออกจากรายการทั้งหมด
-แสดงสถานะออนไลน์/ออฟไลน์บนไอคอนแต่ละแอปพลิเคชันแบบ Real-time
-รองรับการค้นหาแอปพลิเคชันแบบ Instant Search และสลับมุมมองระหว่าง Grid View และ List View

2.1.3  Intranet (อินทราเน็ตคณะ)
-แสดงและจัดกลุ่มประกาศ/ข่าวสารแยกตามประเภทได้อย่างน้อย 4 ประเภท ได้แก่ ด่วน เชิญชวน ประกาศผล และนโยบาย โดยแต่ละประเภทมีสีและไอคอนแตกต่างกัน
-แสดงปฏิทินกิจกรรมในรูปแบบรายเดือน (Monthly Calendar) พร้อม Color Coding แยกประเภทกิจกรรมอย่างน้อย 5 ประเภท ได้แก่ ประชุม สัมมนา สอบ วันหยุด และกำหนดส่ง
-แสดงสถิติองค์กรแบบ Real-time อย่างน้อย 4 รายการ ได้แก่ จำนวนบุคลากร จำนวนหลักสูตร จำนวนผลงานวิจัย และจำนวนนักศึกษาปัจจุบัน
-แสดงข้อมูลติดต่อหน่วยงานภายในคณะ ประกอบด้วยชื่อหน่วยงาน เบอร์โทรศัพท์ อีเมล และสถานที่ตั้ง
-รองรับการ Subscribe ประกาศตามหมวดหมู่หรือฝ่าย และรับการแจ้งเตือนผ่านระบบเมื่อมีประกาศใหม่

2.1.4  Book Meeting (จองห้องประชุม)
-รองรับการจองห้องประชุมและสถานที่ภายในคณะผ่านระบบออนไลน์ โดยกรอกรายละเอียดวัน เวลา หัวข้อการประชุม และจำนวนผู้เข้าร่วม
-แสดงรายการห้องประชุมทั้งหมดพร้อมสถานะว่าง/ไม่ว่างแบบ Real-time
-ระบบป้องกันการจองซ้ำซ้อน (Double Booking) โดยอัตโนมัติ และแจ้งเตือนผู้ใช้ทันทีเมื่อห้องถูกจองแล้ว
-รองรับการยืนยันและยกเลิกการจองผ่านระบบ พร้อมส่งการแจ้งเตือนอัตโนมัติ
-เชื่อมต่อ Microsoft Teams Meeting สำหรับการประชุมออนไลน์ และสามารถดูประวัติการจองย้อนหลังได้

2.1.5  Documents (จัดการเอกสาร)
-รองรับพื้นที่จัดเก็บเอกสาร 3 ระดับ ได้แก่ Central Pool คลังกลางสำหรับเอกสารสาธารณะ Department Pool คลังระดับแผนก และ Personal Pool คลังส่วนตัวสูงสุด 5 GB ต่อคน
-รองรับการอัปโหลดไฟล์จริงหลายรูปแบบพร้อมกัน ได้แก่ PDF, XLSX, PPTX, DOCX, PNG และ JPG
-แสดงรายการเอกสารพร้อมข้อมูลประกอบครบถ้วน ได้แก่ ชื่อไฟล์ ประเภท ขนาด ผู้อัปโหลด และวันเวลาที่แก้ไขล่าสุด
-กำหนดสิทธิ์การเข้าถึงตามระดับ Role โดย Super Admin และ System Admin เข้าถึงได้ทุก Pool, Dept Admin เข้าถึงได้เฉพาะ Central Pool และ Department Pool ของฝ่ายตน, User เข้าถึงได้เฉพาะ Central Pool และ Personal Pool ของตนเอง
-บันทึก Audit Trail ทุกการเข้าถึงและแก้ไขเอกสาร และแสดง Progress Bar การใช้พื้นที่ Storage แบบ Real-time

2.1.6  Projects (การจัดการโครงการ)
-แสดงสถานะโครงการในรูปแบบ Kanban Board แบ่งเป็น 4 คอลัมน์ ได้แก่ Planning, In Progress, Pending Approval และ Completed โดยผู้ใช้สามารถลากและวาง (Drag & Drop) การ์ดระหว่างคอลัมน์ได้
-แสดง Progress Bar เปอร์เซ็นต์ความคืบหน้าของแต่ละโครงการ โดยเจ้าของโครงการหรือ Admin สามารถอัปเดตได้
-สร้างโครงการใหม่ผ่านแบบฟอร์ม ระบุชื่อ ประเภท วัตถุประสงค์ ระยะเวลา และรายชื่อสมาชิก
-รองรับประเภทโครงการอย่างน้อย 6 ประเภท ได้แก่ วิชาการ หลักสูตร สัมมนา วิจัย IT และงบประมาณ
-รองรับการอนุมัติโครงการโดยผู้มีสิทธิ์ระดับ Dept Admin ขึ้นไป พร้อมระบุเหตุผลในการอนุมัติหรือปฏิเสธ

2.1.7  Users & Roles (ผู้ใช้งานและสิทธิ์)
-รองรับบทบาทผู้ใช้งาน 6 ระดับ ได้แก่ Super Admin (สิทธิ์สูงสุด), System Admin (ดูแลระบบ), Dean (คณบดี), Dept Admin (ผู้ดูแลแผนก), User (บุคลากรทั่วไป) และ Viewer (สิทธิ์ดูอย่างเดียว)
-รองรับการซิงค์บัญชีผู้ใช้จาก Active Directory (AD Sync) โดยอัตโนมัติ โดยเมื่อมีการเปลี่ยนแปลงข้อมูลใน AD ระบบต้องอัปเดตให้ตรงกันภายใน 15 นาที
-รองรับการแสดงผลบัญชีผู้ใช้งานได้ไม่น้อยกว่า 300 บัญชีพร้อมกัน โดยไม่มีผลกระทบต่อประสิทธิภาพ
-สามารถเพิ่มผู้ใช้งานใหม่ทั้งแบบ Manual และแบบ Import จากไฟล์ CSV
-กรองผู้ใช้งานตามสถานะได้อย่างน้อย 3 สถานะ ได้แก่ Active, Inactive และ MFA Pending

2.1.8  Audit Log (บันทึกความปลอดภัย)
-บันทึกกิจกรรมแบบ Immutable กล่าวคือข้อมูลที่บันทึกแล้วไม่สามารถแก้ไข ลบ หรือเปลี่ยนแปลงได้ ไม่ว่าจะเป็นผู้ใช้ระดับใดรวมถึง Super Admin
-รองรับประเภทเหตุการณ์อย่างน้อย 8 ประเภท ได้แก่ DOC_UPLOAD, CONFIG_UPDATE, PROJECT_APPROVE, AD_SYNC, USER_LOGIN, USER_LOGIN_FAILED, DASHBOARD_VIEW และ ROLE_CREATE
-แสดงรายละเอียดของแต่ละเหตุการณ์ ประกอบด้วย Timestamp ชื่อผู้กระทำ ประเภทการกระทำ รายละเอียด IP Address และบทบาทของผู้ใช้ขณะนั้น
-สามารถส่งออก (Export) เป็นไฟล์ CSV กรองตามวันที่ ประเภทเหตุการณ์ ชื่อผู้ใช้ และ IP Address และเก็บข้อมูลไม่น้อยกว่า 1 ปี 

2.1.9  System Config (ตั้งค่าระบบ)
-รองรับการตั้งค่า Authentication ได้แก่ Session Timeout , JWT Token Expiry, จำนวนครั้งล็อกอินสูงสุดก่อนล็อกบัญชี และการบังคับใช้ MFA
-รองรับการตั้งค่า LDAP URL สำหรับเชื่อมต่อ Active Directory 
-สามารถปรับแต่ง UI & Branding ได้แก่ Logo ชื่อระบบ และสีธีมหลัก โดยการเปลี่ยนแปลงมีผลทันทีทั้งระบบ
-สามารถกำหนดขนาด Storage สูงสุดต่อผู้ใช้ ประเภทไฟล์ที่อนุญาต และสถานะ/ประเภทโครงการในโมดูล Projects
-รองรับการเชื่อมต่อระบบภายนอกผ่าน API พร้อมจัดการ API Key เข้าถึงได้ตามสิทธิ์ที่กำหนดมี log ในการเข้าถึงข้อมูล


2.2 ข้อกำหนดด้านความมั่นคงปลอดภัยและมาตรฐานระบบ
ระบบยืนยันตัวตนแบบรวมศูนย์ (Single Sign-On) เชื่อมต่อ Active Directory ของมหาวิทยาลัย (LDAP: ldap://ad.tu.ac.th) และซิงค์ข้อมูลอัตโนมัติภายใน 15 นาที
รองรับ Multi-Factor Authentication (MFA) บังคับสำหรับผู้ใช้ระดับ System Admin ขึ้นไปทุกราย
การรับส่งข้อมูลทุกช่องทางเข้ารหัสด้วย TLS/HTTPS
รองรับผู้ใช้งานพร้อมกันไม่น้อยกว่า 300 คน โดยไม่มีผลกระทบต่อประสิทธิภาพ
Uptime ไม่น้อยกว่าร้อยละ 99.5 ต่อปี (SLA)


2.3 สภาพแวดล้อมระบบที่ต้องส่งมอบ
ระบบ TULAW ONE PLATFORM ต้องติดตั้งและให้บริการบน Cloud Server โดยมีข้อกำหนดดังนี้
ผู้รับจ้างต้องจัดหา Cloud Server และดำเนินการติดตั้งระบบบน Cloud Infrastructure ที่มีความมั่นคงปลอดภัยและรองรับการใช้งานตลอด 24 ชั่วโมง 
Cloud Server ต้องตั้งอยู่ในประเทศไทยหรือมีการเก็บข้อมูลในประเทศไทย
ค่าบริการ Cloud Server ตลอดระยะเวลา 1 ปี ต้องรวมอยู่ในวงเงินงบประมาณที่เสนอ โดยผู้รับจ้างต้องระบุรายละเอียดค่าใช้จ่ายรายปีหลังหมดสัญญาในข้อเสนอด้วย


ข้อ 3  ระยะเวลาดำเนินการ
กำหนดระยะเวลาดำเนินการรวม 3 เดือน (90 วัน) นับจากวันลงนามในสัญญา
ข้อ 4  การส่งมอบ
ผู้รับจ้างต้องส่งมอบครบถ้วนตามรายการดังต่อไปนี้
4.1 เอกสารประกอบระบบ
คู่มือผู้ใช้งาน (User Manual) ภาษาไทย ครอบคลุมการใช้งานทุกโมดูลพร้อม Screenshot
คู่มือผู้ดูแลระบบ (Admin Manual) ภาษาไทย ครอบคลุม System Config, Users & Roles และ Audit Log
แผนการบำรุงรักษา (Maintenance Plan) สำหรับดูแลระบบตลอดระยะรับประกัน 1 ปี
4.2 การฝึกอบรม
การฝึกอบรมผู้ดูแลระบบ (Admin Training) ไม่น้อยกว่า 6 ชั่วโมง ครอบคลุม System Config, Users & Roles และ Audit Log
การฝึกอบรมผู้ใช้งานทั่วไป (User Training) ไม่น้อยกว่า 3 ชั่วโมง ครอบคลุมการใช้งานโมดูลหลักทั้งหมด
จัดอบรม ณ สถานที่ของคณะนิติศาสตร์ มหาวิทยาลัยธรรมศาสตร์ หรือรูปแบบออนไลน์ตามที่ตกลงร่วมกัน

เงื่อนไข: ส่งมอบระบบครบทุก 10 โมดูลตามขอบเขตงานข้อ2 พร้อมทดสอบการใช้งานจริง
งวดที่ 3 — ร้อยละ 10 
เงื่อนไข: Go-Live บน Production Cloud Server สำเร็จ ส่งมอบเอกสารประกอบครบถ้วนทุกรายการตามข้อ 5 ผ่านการฝึกอบรมบุคลากร และคณะกรรมการตรวจรับพัสดุ


ข้อ 7  เงื่อนไขอื่น ๆ
7.1 หลักประกันสัญญา
ผู้รับจ้างต้องวางหลักประกันสัญญาร้อยละ 5 ของมูลค่าสัญญา ก่อนลงนามในสัญญา โดยใช้หลักประกันในรูปแบบใดรูปแบบหนึ่ง ดังนี้
เงินสด
เช็คที่ธนาคารสั่งจ่าย ลงวันที่ที่ทำสัญญาหรือก่อนหน้านั้นไม่เกิน 3 วันทำการ
หนังสือค้ำประกันของธนาคารในประเทศตามแบบที่กรมบัญชีกลางกำหนด
หนังสือค้ำประกันของบริษัทประกันภัยที่กรมบัญชีกลางกำหนด
7.2 ค่าปรับ
กรณีผู้รับจ้างส่งมอบงานล่าช้ากว่ากำหนดในสัญญา จะถูกปรับในอัตราร้อยละ 0.2 (ศูนย์จุดสอง) ต่อวัน ของมูลค่างวดที่ส่งมอบล่าช้า นับตั้งแต่วันถัดจากวันกำหนดส่งมอบจนถึงวันที่ส่งมอบจริง
7.3 การรับประกันระบบ
ผู้รับจ้างต้องรับประกันระบบ TULAW ONE PLATFORM เป็นระยะเวลา 1 ปี นับจากวันที่ลงนามรับรองการส่งมอบงานงวดสุดท้าย โดยมีเงื่อนไขดังนี้
แก้ไขข้อบกพร่องหรือ Bug ที่เกิดจากการพัฒนาโดยไม่คิดค่าใช้จ่าย ภายใน 3 วันทำการสำหรับ Critical Bug และภายใน 7 วันทำการสำหรับ Bug ระดับอื่น
ให้บริการ Hotline หรือช่องทางติดต่อ Support ในเวลาราชการ (วันจันทร์–ศุกร์ เวลา 08.30–16.30 น.)
ไม่รวมถึงความเสียหายที่เกิดจากการใช้งานผิดวิธี ภัยธรรมชาติ หรือการแก้ไขระบบโดยบุคคลที่ไม่ได้รับมอบหมายจากผู้รับจ้าง
